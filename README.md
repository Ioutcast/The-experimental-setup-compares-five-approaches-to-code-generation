# Экспериментальный стенд генерации кода

Этот проект является исследовательским стендом для магистерской работы по теме автоматической генерации кода для многослойных приложений.

Стенд написан на Java и Spring Boot. Он не является обычным бизнес-приложением, которое нужно разворачивать для пользователей. Его задача - воспроизвести эксперимент: задать требования к многослойному приложению, прогнать несколько подходов к генерации кода, рассчитать метрики и сформировать таблицы с результатами.

## Что исследуется

В качестве экспериментального объекта используется многослойное workflow-приложение для обработки и согласования заявок. Такая предметная область выбрана потому, что она сложнее простого CRUD-примера: в ней есть жизненный цикл заявки, роли, маршруты согласования, SLA, аудит, события, интеграции и изменение требований.

Эксперимент проверяет, насколько разные подходы генерации способны создавать не только классы сущностей и контроллеры, но и поведение приложения:

- доменную модель;
- workflow и конечный автомат состояний;
- правила построения маршрутов согласования;
- правила доступа;
- SLA и эскалации;
- аудит;
- доменные события;
- интеграционные адаптеры;
- тесты;
- сопровождение при изменении требований.

## Какие подходы сравниваются

В стенде сравниваются пять подходов:

- `Ручной эталон` - условная ручная реализация, которая используется как ориентир качества.
- `Шаблонная генерация` - генерация типовых элементов: сущностей, DTO, репозиториев, контроллеров и базовых сервисов.
- `DSL/model-driven` - генерация на основе формальной спецификации требований.
- `LLM-генерация` - генерация на основе текстовых требований и вызова внешнего генератора через HTTP.
- `Гибридный подход` - сочетание спецификации, шаблонной генерации, LLM-вызова и проверки результата метриками.

## Как устроен эксперимент

Эксперимент состоит из десяти версий требований:


| Версия | Смысл изменения                                            |
| ------ | ---------------------------------------------------------- |
| `v1`   | Базовый процесс согласования заявки                        |
| `v2`   | Добавление SLA и эскалаций                                 |
| `v3`   | Добавление службы безопасности                             |
| `v4`   | Добавление юридического согласования                       |
| `v5`   | Добавление делегирования согласующих                       |
| `v6`   | Добавление внешней финансовой системы                      |
| `v7`   | Добавление многофилиальной структуры организации           |
| `v8`   | Добавление настраиваемых типов заявок                      |
| `v9`   | Добавление аудита запрещенных действий                     |
| `v10`  | Добавление повторной генерации без потери ручных изменений |


Для каждой версии требований стенд запускает все пять подходов. В итоге получается 50 экспериментальных запусков: 10 версий требований умножаются на 5 подходов.

Для каждого запуска формируются сгенерированные артефакты и рассчитываются метрики.

## Какие метрики считаются

Стенд рассчитывает несколько групп метрик.

Функциональные метрики показывают, насколько подход покрыл заданное поведение:

- доля реализованных сценариев;
- корректность workflow-переходов;
- корректность правил согласования;
- корректность правил доступа;
- корректность SLA-сценариев.

Архитектурные метрики показывают, сохраняется ли многослойная структура:

- количество нарушений границ слоев;
- корректность направления зависимостей;
- тонкость контроллеров;
- чистота domain-слоя.

Метрики генерации показывают трудоемкость получения результата:

- объем сгенерированного кода;
- объем ручных исправлений;
- успешность повторной генерации;
- число итераций запросов;
- покрытие спецификации.

Метрики тестирования показывают качество проверок:

- доля проходящих тестов;
- доля проходящих тестов бизнес-правил;
- доля проходящих workflow-тестов;
- mutation score;
- число регрессий.

Метрики сопровождаемости показывают, насколько сложно менять требования:

- трудоемкость изменения;
- количество измененных файлов;
- доля сгенерированных изменений;
- число конфликтов ручных правок;
- время адаптации.

## Структура проекта

Основной код находится в `src/main/java/ru/master/exp/`.

Главные файлы:

- `CodeGenerationExperimentApplication.java` - точка входа Spring Boot приложения. При запуске создает спецификацию, стратегии генерации, оцениватель и записывает результаты.
- `ExperimentCore.java` - маркерный класс модуля эксперимента. Основная логика разнесена по отдельным файлам.
- `ExperimentModel.java` - модели данных эксперимента: подходы, слои, требования, сценарии, артефакты и метрики.
- `ExperimentSpecFactory.java` - входная спецификация требований `v1-v10`.
- `GenerationStrategies.java` - стратегии ручного эталона, шаблонной, DSL, LLM и гибридной генерации.
- `ModelClients.java` - HTTP-клиент внешнего генератора и локальный fallback-клиент.
- `GeneratedArtifactFactory.java` - создание демонстрационных артефактов генерации.
- `ExperimentEvaluator.java` - расчет метрик.
- `ExperimentRunner.java` - запуск полного цикла эксперимента.
- `ReportWriter.java` - запись CSV, Markdown-таблиц и артефактов.
- `baseline/` - ручной эталон ключевых правил, разнесенный по отдельным классам: заявка, статусы, роли, конечный автомат, маршруты согласования, права доступа и аудит.
- `ExperimentCoreTest.java` - тесты, проверяющие создание спецификации и формирование отчетов.
- `application.yml` - настройки запуска, директория результатов и параметры HTTP-вызова внешнего генератора.

Проектные настройки Maven находятся в `pom.xml`. Проект адаптирован под Java 11 и Spring Boot 2.7, чтобы запускаться на установленной локальной версии JDK.

## Где находится ручной эталон

Ручной эталон находится в пакете `src/main/java/ru/master/exp/baseline/`.

Он нужен как контрольная реализация ключевых правил предметной области. Эталон разнесен по отдельным файлам:

- `ApprovalRequest.java` - заявка и ее основные поля;
- `RequestStatus.java` - статусы жизненного цикла заявки;
- `Role.java` - роли пользователей;
- `Decision.java` - возможные решения согласующего;
- `RequestStateMachine.java` - допустимые переходы конечного автомата;
- `ApprovalRouteBuilder.java` - построение маршрута согласования по сумме, персональным данным и договорному признаку;
- `AccessPolicy.java` - проверки прав автора, руководителя, согласующего и администратора;
- `AuditService.java` и `AuditRecord.java` - аудит изменения статуса и запрещенных действий.

Файл `ManualBaseline.java` оставлен как маркерный класс пакета. Он показывает, что весь пакет относится к ручному эталону, но сама логика находится в отдельных классах.

В экспериментальном цикле ручной эталон также представлен стратегией `ManualBaselineGenerationStrategy` в файле `src/main/java/ru/master/exp/GenerationStrategies.java`. Эта стратегия участвует в общем сравнении вместе с шаблонным, DSL, LLM и гибридным подходами.

После запуска результаты ручного эталона можно посмотреть в `results/generated-artifacts/manual_baseline/` и в таблицах `results/tables-for-thesis.md`.

## Как запустить с нуля

Нужны Java 11 и Maven.

Проверить версии:

```bash
java -version
mvn -version
```

Запустить тесты:

```bash
mvn test
```

Запустить эксперимент:

```bash
mvn spring-boot:run
```

После запуска в консоли должно появиться примерно такое сообщение:

```text
Experiment completed
Requirement versions: 10
Approaches: 5
Metric rows: 50
Main report: ...Masters_Exp\results\tables-for-thesis.md
```

Это означает, что эксперимент выполнен для десяти версий требований и пяти подходов.

## Где смотреть результаты

После запуска создается папка `results/`.

Основные результаты:

- `results/tables-for-thesis.md` - результаты;
- `results/metrics/generation-results.csv` - полная таблица метрик по всем 50 запускам;
- `results/metrics/evolution-results.csv` - результаты по версиям требований;
- `results/metrics/generation-run-log.txt` - журнал запусков генерации;
- `results/generated-artifacts/` - сгенерированные файлы по подходам и версиям требований.

Файл `results/tables-for-thesis.md` удобен для диплома: в нем уже собраны сводные таблицы по подходам, эволюции требований, архитектурной устойчивости, трудоемкости изменений и интерпретации результатов.

CSV-файлы можно открыть в Excel, LibreOffice Calc или импортировать в Python/R для построения графиков.

## Как проводился эксперимент

Порядок проведения эксперимента следующий:

1. В коде формируется единая спецификация экспериментального приложения.
2. В спецификации задаются десять версий требований `v1-v10`.
3. Для каждой версии требований последовательно запускаются пять подходов генерации.
4. Каждый подход формирует набор условно сгенерированных артефактов по слоям приложения.
5. Для каждого результата рассчитываются функциональные, архитектурные, тестовые и сопровождаемые метрики.
6. Итоговые значения записываются в CSV-файлы.
7. На основе CSV-данных формируется Markdown-отчет с таблицами

Важно: стенд моделирует исследовательский эксперимент, а не заменяет промышленный генератор. Его назначение - сделать сравнение подходов воспроизводимым, измеримым и удобным для описания в дипломной работе.

## Вызов внешнего генератора по IP

В проекте предусмотрен HTTP-клиент для вызова внешнего генератора.

Адрес задается через переменную окружения:

```bash
MODEL_API_BASE_URL=
```

Клиент отправляет запрос:

```text
POST /generate
```

Тело запроса содержит JSON-поля:

```json
{
  "approach": "LLM",
  "version": "v1",
  "requirements": ["..."],
  "layers": ["domain", "workflow", "policy"]
}
```

Ожидаемый ответ:

```json
{
  "content": "generated code or generation plan",
  "confidence": 0.72
}
```

Если адрес не задан или внешний сервис недоступен, используется локальный детерминированный fallback. Это нужно для того, чтобы эксперимент можно было запустить на любом компьютере и получить таблицы без внешней зависимости.
